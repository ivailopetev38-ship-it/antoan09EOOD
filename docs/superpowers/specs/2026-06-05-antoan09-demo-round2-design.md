# Антоан-09 — Демо-кръг 2: AI стикер, въвеждане, напомняния, PDF, импорт

**Дата:** 2026-06-05
**Статус:** одобрен дизайн (чака преглед на spec)
**Тип:** платено демо (за продажба на пълния проект 1800€)

## 1. Контекст

Демото с базовия скелет вече е live (нормативен двигател, протокол 1:1 Word,
дашборд, QR карта/скенер, календар, Hermes AI чат върху данните в Telegram).
Този кръг добавя най-силните липсващи функции от ТЗ-то, за да убеди клиента да
поръча пълния проект. Целта е **demo-grade, но с реални, преизползваеми функции** —
без код за изхвърляне.

Източник на истината: двата имейла на клиента (основно ТЗ `19e74ce4cc329119` +
стикер-уточнение `19e7511da579678e`). Няма нови/обновени задания.

## 2. Обхват

**Включено (този кръг):**
1. Данни: пълна номенклатура от ТЗ + реалистичен демо-набор бройки + **импорт-механизъм**.
2. **AI стикер → протокол**: снимка (на сайта и на картата/линка) → разпознаване → статус → попълнен протокол. + ръчен избор като алтернатива.
3. **Минимално въвеждане**: добави обект/гасител; „запиши извършена услуга" (обновява история + срокове).
4. **Демо напомняния**: екран + изпращане (през Hermes/Telegram, с fallback преглед).
5. **PDF изход** на протокола (освен Word).

**Извън обхвата (за пълния проект 1800€):** автентикация/роли, връзка със склад,
оптимизация на маршрути, многоезичен чат, мобилно нативно приложение.

## 3. Водещи принципи

- **Без OpenAI ключ в приложението.** Целият AI минава през **Hermes** (който вече
  има LLM провайдър на сървъра). Приложението вика абстрактен провайдър по HTTP.
- **Провайдър-агностичност + безопасен fallback:** всяка външна AI/комуникационна
  зависимост има интерфейс + демо-fallback, така че целият поток върви на екран и
  преди Hermes-крайът да е готов.
- **Авто-генерирай от наличните данни; питай само при липса/съмнение.**
- Преизползваме съществуващото: `computeExtinguisherStatus` (двигател),
  `buildProtocol` (сглобяване на протокол), `getEnrichedExtinguishers` (данни).

## 4. Компонент 1 — Данни и импорт

### 4.1 Номенклатура (reference данни)
Нови/допълнени таблици, seed-нати от ТЗ (всичко разширяемо):
- `brands` — 17 марки (Солти, Огнехром, Дрипалдер, Ятрус, Торнадо, Sparky, Gloria,
  Bavaria, Total, Ceasefire, Minimax, Tyco, Sicli, Chubb, FirePro, Ansul, Kidde).
- `agents` — гасителни вещества (ABC прах напр. „Кобра ABC 50"/„Верея ABC 40",
  пенообразувател напр. „Щамекс FF", вода, CO₂).
- Типове и капацитети: фиксиран, но пълен набор (валидиран списък), използван в менютата:
  - прахов ABC/BC: 1, 2, 6, 12, 25, 50 кг
  - воден: 9 л · водопенен: 9 л · CO₂: 2, 5 кг

### 4.2 Бройки (units)
- Засега: реалистичен демо-набор (замества текущия seed) с разнообразни статуси
  (ОК / скоро / просрочен / БРАК), покриващ всички типове.
- Реалните бройки идват по-късно от клиента → качват се през импорта (виж 4.3).

### 4.3 Импорт-механизъм
- Админ страница `/admin/import`: поставяш таблица (CSV/TSV; колони: клиент, обект,
  марка, модел, сериен №, година, тип, капацитет, последно ТО, последно ПЗ,
  последно ХИ, щампа-година, забележки).
- `POST /api/import` → парсва → upsert на клиенти/обекти/гасители + начални
  `service_events`. Идемпотентно по сериен №.
- Demo-grade: без сложна валидация; ясни грешки на ред.

## 5. Компонент 2 — AI стикер → протокол (главната wow-функция)

### 5.1 Поток (уеб)
1. Качване на снимка: разширяваме `/skan` (добавяме снимка/файл към сегашния QR скенер) **и** бутон на картата `/pg/[id]`.
2. `POST /api/vision/sticker` (multipart/base64) → качва изображението в Supabase
   Storage (bucket `stickers`) → взима **подписан URL** (временен).
3. Вика `recognizeSticker(imageUrl)` (провайдър — виж 5.3).
4. Полученият JSON се мапва към гасител (намери по сериен № или предложи нов) →
   `computeExtinguisherStatus` → статус (ТО/ПЗ/ХИ/БРАК).
5. Екран „Преглед": разпознати полета (редактируеми) + статус + бутон
   **Word** / **PDF** (през `buildProtocol`).

### 5.2 Ръчен избор (алтернатива без снимка)
- Същият екран предлага „Избери ръчно": обект → гасител → същият преглед/протокол.

### 5.3 Vision провайдър (през Hermes, без OpenAI ключ)
Интерфейс `VisionProvider.recognize(imageUrl): Promise<StickerFields>`.

**Адаптер „hermes"** (по подразбиране, когато е зададен `HERMES_VISION_URL`):
```
POST {HERMES_VISION_URL}
Authorization: Bearer {HERMES_API_TOKEN}
Content-Type: application/json
{ "imageUrl": "<подписан URL>", "task": "extinguisher_sticker", "schemaVersion": 1 }
→ 200 { "ok": true, "fields": StickerFields, "confidence": 0..1, "raw": "<текст>" }
```
`StickerFields` (всички полета nullable):
`{ brand, model, serial_number, manufacture_year, type, capacity, agent,
   stamps: [{ kind: "TO|recharge|HI", date: "YYYY-MM-DD" }], scrap_year }`

**Адаптер „fallback"** (когато `HERMES_VISION_URL` липсва): връща детерминиран мок от
вграден примерен стикер + флаг `demo: true`. UI показва бележка „демо разпознаване
(Hermes още не е вързан)". Така целият поток работи на екран веднага.

**Договор за Hermes-крайът (сървърна работа, по-късно):** малък HTTP endpoint, който
приема снимка (по URL), пуска vision на агента и връща `StickerFields`. Това е
„после ще кажем на Hermes какво да прави". До тогава работи fallback-ът.

## 6. Компонент 3 — Минимално въвеждане (живо)

- Форми (уеб): „Нов обект", „Нов гасител" (към обект), „Запиши услуга".
- „Запиши услуга": дейност (ТО/ПЗ/смяна прах/смяна пяна/ХИ) + дата + техник + стикер №
  (+ гасително вещество — задължително при ПЗ, скрито при ТО, по ТЗ т.7).
- `POST /api/service-events` → запис → историята и **следващите срокове се
  преизчисляват автоматично** от двигателя (без ръчно въвеждане на падежи).

## 7. Компонент 4 — Демо напомняния

- Екран `/napomnyania`: клиенти/обекти с предстоящ/просрочен падеж (преизползва
  `getSchedule`/due заявките) + предложен текст на напомнянето.
- Бутон „Изпрати напомняне" → `POST /api/reminders/send`.
- **Notify провайдър** (същия модел като vision): адаптер „hermes" праща през
  Telegram (Hermes владее бота) при зададен `HERMES_NOTIFY_URL`; fallback = връща
  готовия текст за преглед на екран. За демо preview-ът винаги работи; реалното
  пращане е „флип" щом Hermes-крайът е готов.

## 8. Компонент 5 — PDF изход

- `buildProtocol` вече дава структурирани данни. Добавяме PDF рендер със
  **server-side React-PDF** (`@react-pdf/renderer`, надежден на Vercel), огледален на
  Приложение №9 (същите 14 полета + ред на гасителите).
- `GET /api/protocols/generate?format=pdf` (или отделен route) → връща `.pdf`.
- Word остава както е; PDF е допълнителен бутон.

## 9. Схема (Supabase) — промени

- Нови таблици: `brands`, `agents` (seed от ТЗ). По избор `models` (иначе свободен текст).
- Storage bucket `stickers` (за качените снимки; подписани URL-и).
- Без промени по RLS логиката за демото (service_role на сървъра, както сега).

## 10. Env / деплой

- Съществуващи: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`,
  `SUPABASE_SERVICE_ROLE_KEY`, `HERMES_API_TOKEN`.
- Нови (по избор, активират реалния Hermes път; без тях работят fallback-ите):
  - `HERMES_VISION_URL` — Hermes vision endpoint.
  - `HERMES_NOTIFY_URL` — Hermes Telegram-notify endpoint.
- **Без `OPENAI_API_KEY` в приложението.**
- Деплой: GitHub push → Vercel (както сега). Промяна на env → redeploy (внимание:
  CRLF при CLI — добавяй стойности чисто).

## 11. Тестване

- Чисти функции (Vitest): парсване на импорт-редове; мапване `StickerFields` → вход
  за двигателя; избор на дейност спрямо статус; PDF данни = Word данни.
- Двигателят и протоколът са вече покрити с тестове — преизползват се.
- E2E ръчно: снимка (fallback) → преглед → Word+PDF; запис на услуга → промяна на
  статус/срок; импорт на примерен CSV; напомняне (preview).

## 12. Зависимости / отворени

- **Реалната база с бройки** — чака се от клиента; качва се през импорта.
- **Hermes vision/notify endpoint-и** — сървърна работа (Codex/Hermes), по договора
  в 5.3 / 7. До тогава: fallback-и.
- Подреждане на изпълнението — в плана (writing-plans), фазирано и тествано.

## 13. Принцип за реда на работа

Приоритет по demo-въздействие: (1) Данни+импорт → (2) AI стикер→протокол (с fallback)
→ (3) Въвеждане → (4) PDF → (5) Напомняния. Всяка част е самостоятелна и деплойваема.
